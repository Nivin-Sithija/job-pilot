- **First real production deploy (2026-06-21) surfaced four distinct bugs, all found by reproducing the actual failure on the live droplet, none guessed**:
  - **Homepage 500 in production**: root cause was `NEXT_PUBLIC_*` vars getting inlined into the JS bundle at `next build` time, including in code that *runs on the server* (not just client components) — the droplet's container had been started with these vars only set at `docker run` time (too late, bundle already compiled empty). Fixed by always passing them as Docker `--build-arg`s (see `code-standards.md`'s Environment Variables section, which now documents this build-time/runtime split explicitly).
  - **OAuth post-login redirect went to an unreachable internal hostname** (`http://<container-id>:3000/...`, `DNS_PROBE_FINISHED_NXDOMAIN`): Next.js 16 standalone mode's generated `server.js` always passes Docker's auto-injected `HOSTNAME` env var as an explicit server `hostname`, which makes Next.js build `request.url` from that internal hostname instead of the proxied request's real `Host` header — confirmed by tracing into `node_modules/next/dist/esm/server/{base-server,next-server}.js`. Any `new URL(path, request.url)` redirect leaks this behind a reverse proxy. Fixed in `app/api/auth/callback/route.ts` by building redirects from `NEXT_PUBLIC_APP_URL` instead, matching the pattern `actions/auth.ts` already used correctly.
  - **CSP blocked PostHog's Google Fonts load** (`rrweb-record.js`, its session-recording overlay, loads Mulish from `fonts.googleapis.com`/`fonts.gstatic.com` independent of the app's own self-hosted `next/font` Inter) — fixed by adding both origins to `next.config.ts`'s `style-src`/`font-src`.
  - **OAuth login silently bounced back to `/login` with no error param** — two-layer cookie bug, both confirmed by reading compiled server bundles directly on the droplet (`docker exec ... grep` against `.next/server/chunks/`) rather than guessing: (1) our own PKCE code-verifier cookie in `actions/auth.ts` hardcoded `secure: true`, dropped by the browser on the droplet's plain-HTTP origin — fixed by deriving it from `NEXT_PUBLIC_APP_URL`'s protocol (`IS_HTTPS_APP_URL`); (2) even after that fix, the **InsForge SDK's own session cookies** (`accessToken`/`refreshToken`, set inside `exchangeOAuthCode`) stamp `secure: true` whenever `NODE_ENV === "production"` (`@insforge/sdk/dist/ssr/middleware.js`) — not configurable from app code at all. Confirmed by `grep`-ing the compiled bundle for the literal SDK logic. Since `NODE_ENV` is always `production` in the Docker image, the only real fix was serving real HTTPS. Solved by pointing Caddy at a free `nip.io` hostname (`64.227.162.172.nip.io` — resolves straight to the droplet IP, no domain purchase needed) so it auto-provisions a genuine Let's Encrypt cert; `DOMAIN` in `.env.production` and the `NEXT_PUBLIC_APP_URL` GitHub Actions variable were both updated to match. Verified via `openssl s_client` (real Let's Encrypt issuer, not self-signed) and a full HTTP→HTTPS→sign-in round trip.
  - General lesson applied throughout this debugging session: every claim was verified against the actual running container (`docker exec`, `docker compose logs`, `grep`-ing compiled bundles) or actual Next.js/SDK source in `node_modules`, never assumed from documentation or training data — consistent with this project's existing "read the real installed source, not the docs" precedent (see Features 04/06/07 above).
